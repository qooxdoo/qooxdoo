[k]
This is to test the default demo of a contribution skeleton. After creating the skeleton, change dir to trunk/demo/default. If you're not testing in a qooxdoo-contrib checkout, you need to adapt QOOXDOO_PATH in config.json. Then, run the normal tests.
[l]
  • Simple migration test:
    • create a skeleton application from a previous release
    • create a corresponding skeleton from the current release
    • delete the source/class folder from the current skeleton
    • copy over the source/class folder from the previous skeleton
    • run generate.py migration
  • Advanced migration test:
    • copy the Feedreader application of a previous release
    • edit config.json/QOOXDOO_PATH to point to the qooxdoo installation under test
    • run generate.py migration
In both cases, the migration job should terminate normally, with various log messages on the console. During migration you should be offered to enter your "old" qooxdoo version, an a suitable upgrade path leading up to the version under test should be printed out. After the migration, a file migration.log should be created that contains the per-version, per-file entries, deprecation warnings and modification statements. After that, a generate.py source may or may not terminate normally, and the created application, if any, may or may not run in the browser.
