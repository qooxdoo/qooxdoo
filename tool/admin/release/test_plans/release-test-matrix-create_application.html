Annotations

General
The main objective of these tests is to make sure the generator jobs run through, and produce applications that load ok in the browser.
[a]
The console output should contain no warnings.

[b]
To verify pretty-printing, look at some code (e.g. Application.js, DemoTest.js) including its comments, just enough to make sure it is not wildly misformatted. (Don't count indent levels or ponder about opening curlies). Then run a subsequent source job, to make sure it still compiles.
[c]
Without any generation, open the local index.html. The demo should run immediately.
[d]
The job to run is build-min (or just generate.py without a job). It will re-create the Website library into the script/ folder. Reload the index.html to ensure it is working.

[i]
How to set up the Simulator environment:
  • Download the following files and place them in an easily accessible folder, e.g. /home/affe/workspace/simulator:
    • Rhino
    • Selenium Java Client
    • Selenium Server
  • Open a shell and go to the directory where you downloaded the files, then run this command: 
    java -jar selenium-server.jar -userExtensions <QOOXDOO_PATH>/component/simulator/tool/user-extensions/user-extensions.js
    • QOOXDOO_PATH must be replaced with the (relative or absolute) file system path of a qooxdoo SDK or trunk checkout.
    • The Selenium server should now run on localhost:4444. Do not close the shell window.
  • Create a qooxdoo Desktop skeleton application, e.g. myapplication, and generate the build version.
  • Make sure the application can be accessed over HTTP, e.g. by running python -m CGIHTTPServer
  • Edit the config.json file of the application.
    • Add the following job configuration: 
          "simulation-run" :
          {
            "let" :
            {
              "SIMULATOR_CLASSPATH" : [ "/home/affe/workspace/simulator/selenium-java-client-driver.jar",
                                        "/home/affe/workspace/simulator/js.jar" ]
            },
            
            "environment" :
            {
              "simulator.selServer" : "localhost",
              "simulator.selPort" : 4444,
              "simulator.testBrowser" : "*firefox",
              "simulator.autHost" : "http://localhost",
              "simulator.autPath" : "/home/affe/workspace/myapplication/build/index.html"
            }
          }
    • Use the file system path of the directory where you downloaded the files earlier for the two entries in SIMULATOR_CLASSPATH
    • Replace the value of simulator.autPath with the web server path of your application
  • Build and run the Simulator application
    • In your application skeleton, run 
      generate.py simulation-build
      and note any warnings (see [a])
    • Provided there were no errors, run 
      generate.py simulation-run
    • Two Firefox windows should open and the test should run. The Generator's shell output should list two tests as passed.
    • If the browser could not be located, add the direct path behind *firefox seperated by a space.
[k]
This is to test the default demo of a contribution skeleton. After creating the skeleton, change dir to trunk/demo/default. If you're not testing in a qooxdoo-contrib checkout, you need to adapt QOOXDOO_PATH in config.json. Then, run the normal tests.
[l]
  • Simple migration test:
    • create a skeleton application from a previous release
    • create a corresponding skeleton from the current release
    • delete the source/class folder from the current skeleton
    • copy over the source/class folder from the previous skeleton
    • run generate.py migration
  • Advanced migration test:
    • copy the Feedreader application of a previous release
    • edit config.json/QOOXDOO_PATH to point to the qooxdoo installation under test
    • run generate.py migration
In both cases, the migration job should terminate normally, with various log messages on the console. During migration you should be offered to enter your "old" qooxdoo version, an a suitable upgrade path leading up to the version under test should be printed out. After the migration, a file migration.log should be created that contains the per-version, per-file entries, deprecation warnings and modification statements. After that, a generate.py source may or may not terminate normally, and the created application, if any, may or may not run in the browser.
[m]
Testing on a single platform is enough.
