General
The main objective of these tests is to make sure the generator jobs run through, and produce applications that load ok in the browser.
[a]
The console output should contain no warnings.

[b]
To verify pretty-printing, look at some code (e.g. Application.js, DemoTest.js) including its comments, just enough to make sure it is not wildly misformatted. (Don't count indent levels or ponder about opening curlies). Then run a subsequent source job, to make sure it still compiles.
[c]
Without any generation, open the local index.html. The demo should run immediately.
[d]
The job to run is build (or just generate.py without a job). It will re-create the Website library into the script/ folder. Reload the index.html to ensure it is working.
[k]
This is to test the default demo of a contribution skeleton. After creating the skeleton, change dir to trunk/demo/default. If you're not testing in a qooxdoo-contrib checkout, you need to adapt QOOXDOO_PATH in config.json. Then, run the normal tests. The grunt toolchain won't work here - use the Generator.
[l]
  • Simple migration test:
    • create a skeleton application from a previous release
    • create a corresponding skeleton from the current release
    • delete the source/class folder from the current skeleton
    • copy over the source/class folder from the previous skeleton
    • run generate.py migration
  • Advanced migration test:
    • copy the Feedreader application of a previous release
    • edit config.json/QOOXDOO_PATH to point to the qooxdoo installation under test
    • run generate.py migration
In both cases, the migration job should terminate normally, with various log messages on the console. During migration you should be offered to enter your "old" qooxdoo version, a suitable upgrade path leading up to the version under test should be printed out. After the migration, a file migration.log should be created that contains the per-version, per-file entries, deprecation warnings and modification statements. After that, a generate.py source may or may not terminate normally, and the created application, if any, may or may not run in the browser.
[m]
Testing on a single platform is enough.
[n]
Go to tool/grunt/task/package/{packageName} run npm install and afterwards run the npm cmd under test there.
[o]
Open up the generated dir (e.g. api or test/data/myapp) or app in browser and check if everything works as expected (interact/click through some pages).
[p]
If not happend before:
  1. cd tool/grunt
  2. ./setup.js
Then:
  1. cd tool/grunt/{taskName}
  2. run task under test (grunt {taskName})
  3. test task under test (e.g. open browser for source/build version - see [o])
[q]
grunt clean:app is the equivalent to ./generate.py clean and grunt clean:cache is the equivalent to ./generate.py distclean.
